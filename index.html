Module A — Terminal & Command Line
A1
Terminal là gì và tại sao cần dùng?
Khái niệm nền
▼
Analogy cho designerTerminal như Figma mà không có GUI — thay vì click vào nút, bạn gõ tên nút. Chậm hơn lúc đầu, nhưng nhanh hơn nhiều khi đã quen và có thể tự động hóa mọi thứ.
Sơ đồ — GUI vs CLI
Nội dung chính
  • Terminal (CLI) = giao diện text để ra lệnh trực tiếp cho máy tính — không dùng chuột.
  • Shell = chương trình đọc lệnh bạn gõ và chuyển cho OS thực thi. macOS/Linux dùng zsh hoặc bash.
  • Vibe coder cần terminal vì: Claude Code chạy trong terminal, Git dùng terminal, npm install cần terminal.
  • Cấu trúc lệnh: git commit -m "message" → git là lệnh, commit là hành động, -m là tùy chọn, "message" là đối số.
  • Khi thấy error đỏ trong terminal: đọc dòng cuối cùng trước — thường là dòng quan trọng nhất.
Bỏ qua đượcShell scripting, viết alias, cấu hình .zshrc sâu.

Module B — Cách Web Hoạt Động
B1
Client, Server và HTTP
Khái niệm nền
▼
Analogy cho designerClient như khách gọi món (gửi request). Server như nhà bếp (xử lý và trả response). HTTP là ngôn ngữ chung để gọi món — cả hai phải dùng đúng format mới hiểu nhau.
Sơ đồ — HTTP Request / Response Cycle
Nội dung chính
  • Client = thiết bị/phần mềm gửi yêu cầu — thường là trình duyệt của bạn.
  • Server = máy tính nhận yêu cầu, xử lý, và trả kết quả về.
  • HTTP = tập quy tắc định dạng request và response. HTTPS = HTTP + mã hóa SSL.
  • Request có method: GET (lấy dữ liệu), POST (gửi dữ liệu), PUT/DELETE (sửa/xóa).
  • Response trả về status code: 200 = OK, 404 = không tìm thấy, 500 = lỗi server.
DesignGurus — What is HTTP and how do HTTP requests work? Fastly — HTTP Requests Explained
B2
URL, Domain và DNS
Khái niệm nền
▼
Analogy cho designerDomain như tên thương hiệu (nike.com). IP address như địa chỉ nhà thật (104.26.2.33). DNS như danh bạ điện thoại — bạn gõ tên, nó tra ra số.
Giải phẫu một URL
DNS lookup — Từ tên domain đến IP
Nội dung chính
  • localhost = 127.0.0.1 = chính máy bạn. Dev server chạy ở đây trước khi deploy.
  • Port = "cửa" phân biệt service. :3000, :8080 là port. HTTP mặc định dùng 80, HTTPS dùng 443.
  • Khi deploy lên Vercel + trỏ domain: bạn đang thêm DNS record trỏ tên → IP của Vercel.
  • DNS propagation = thời gian thay đổi lan ra toàn cầu — thường 5 phút đến 48 giờ.
Bỏ qua đượcDNS propagation chi tiết, TTL, nameserver tự quản lý.

Module C — JavaScript & Node.js ở Mức Đọc Hiểu
C1
Node.js là gì và tại sao cần biết?
Khái niệm nền
▼
Analogy cho designerJavaScript trong trình duyệt như Figma chạy trên máy bạn. Node.js như Figma chạy trên server — cùng ngôn ngữ thiết kế nhưng môi trường khác, dùng được cho tác vụ khác.
JavaScript — Browser vs Node.js
Nội dung chính
  • JavaScript ban đầu chỉ chạy trong trình duyệt. Node.js cho phép JavaScript chạy trên server, ngoài trình duyệt.
  • Gần như toàn bộ tool frontend hiện đại (Next.js, Vite, Tailwind CLI) đều chạy trên Node.js.
  • Bạn cần Node.js cài trên máy để chạy npm install, npm run dev.
  • Runtime = môi trường thực thi code. Node.js là runtime của JavaScript ngoài browser.
  • Khi bạn đóng terminal, dev server chết vì process bị tắt — đó là lý do cần deploy lên Vercel để chạy 24/7.
Bỏ qua đượcEvent loop, non-blocking I/O, viết server Node.js từ đầu.
Educative — What is Node.js? DigitalOcean — How to use Node.js modules with npm
C2
package.json là gì?
Thực hành đọc hiểu
▼
Analogy cho designerpackage.json như file manifest.json của Figma Plugin — khai báo tên, version, dependencies, và các lệnh có thể chạy. Mở file này ra là biết project đang dùng gì.
Ví dụ đọc hiểu
{
  "name": "cynapze",
  "scripts": {
    "dev":   "next dev",     ← npm run dev   → chạy dev server
    "build": "next build",  ← npm run build → build production
    "start": "next start"   ← npm run start → chạy sau khi build
  },
  "dependencies": {
    "next":  "15.0.0",       ← Next.js version 15
    "react": "^19.0.0"      ← React v19 trở lên
  },
  "devDependencies": {
    "tailwindcss": "^4.0"  ← chỉ dùng khi dev
  }
}
Nội dung chính
  • scripts = lệnh shortcut. npm run dev chạy lệnh tương ứng trong scripts.
  • dependencies = packages cần thiết để app chạy trên production.
  • devDependencies = packages chỉ cần khi dev, không cần trên production.
  • npm install đọc file này và tải tất cả packages xuống node_modules/.
  • Claude Code tự cập nhật package.json và chạy install khi thêm package mới.
Bỏ qua đượcSemantic versioning chi tiết, peerDependencies, publish package lên npm.
GeeksforGeeks — Node.js package.json NodeSource — The Basics of package.json

Module D — Networking Cơ Bản
D1
Domain hoạt động khi deploy như thế nào?
Khái niệm mở rộng
▼
Analogy cho designerMua domain như mua tên thương hiệu. Trỏ DNS về Vercel như đặt biển tên thương hiệu trước cửa hàng của Vercel. Khách chỉ cần nhớ tên thương hiệu, không cần nhớ địa chỉ thật.
Flow — Mua domain → Deploy → Trỏ DNS
Nội dung chính
  • A record = trỏ domain → IP address cụ thể.
  • CNAME record = trỏ domain → domain khác (Vercel dùng cách này).
  • DNS propagation = thời gian để thay đổi lan ra toàn cầu — thường 5 phút đến 48 giờ.
  • Subdomain (app.mysite.com) cần DNS record riêng — Vercel tự tạo khi bạn thêm custom domain.

D2
Process, Runtime và tại sao app cần "chạy"?
Khái niệm mở rộng
▼
Analogy cho designerFile Figma không tự mở — bạn phải bấm "Open". App web cũng vậy — cần có process đang chạy mới serve được trang. Static site = file PDF đã export — không cần Figma mở mới đọc được.
Static site vs Server-side app
Nội dung chính
  • Process = một chương trình đang thực thi trong bộ nhớ. npm run dev tạo ra 1 process.
  • Khi bạn đóng terminal → process chết → localhost:3000 không còn respond.
  • Vercel giữ process chạy 24/7 — hoặc dùng serverless (bật khi có request, tắt sau đó).
  • Cold start = lần đầu gọi serverless hơi chậm vì cần "khởi động" — các lần sau nhanh hơn.
Bỏ qua đượcThread, memory management, process manager (PM2), daemon.

Module E — Kiến Trúc Hệ Thống Web
E1
Frontend · Backend · Database — 3 tầng cơ bản
Khái niệm nền
▼
Analogy cho designerFrontend như màn hình Figma bạn thấy — đẹp, có tương tác. Backend như plugin logic chạy ngầm — xử lý dữ liệu, thực thi rule. Database như Figma file được lưu trên cloud — nơi mọi thứ được ghi nhớ lâu dài.
Sơ đồ — 3 tầng kiến trúc web
Nội dung chính
  • Frontend = tầng giao diện — mọi thứ user nhìn thấy và tương tác. Chạy trên browser của người dùng.
  • Backend = tầng logic — xử lý yêu cầu, kiểm tra quyền, tính toán. Chạy trên server.
  • Database = tầng dữ liệu — lưu trữ mọi thứ lâu dài (user, post, order). Backend giao tiếp với DB, không phải frontend.
  • Quy tắc vàng: Frontend → Backend → Database. Frontend không bao giờ truy cập DB trực tiếp — đây là lỗ hổng bảo mật.
  • Trong Next.js: folder app/ = frontend, folder app/api/ = backend (serverless functions), Supabase/PlanetScale = database.
Bỏ qua đượcMicroservices, message queue, load balancer, database sharding.

E2
API là gì và tại sao nó ở khắp nơi?
Khái niệm nền
▼
Analogy cho designerAPI như menu nhà hàng — bạn không vào bếp tự lấy đồ ăn, bạn gọi theo menu. Bếp (backend) làm theo yêu cầu đúng format trong menu, trả ra đúng món. Bạn không cần biết bếp nấu thế nào.
API request/response — ví dụ thực tế
Nội dung chính
  • API (Application Programming Interface) = tập quy tắc cho phép 2 phần mềm nói chuyện với nhau.
  • REST API = loại API phổ biến nhất — dùng HTTP method (GET, POST, PUT, DELETE) và trả về JSON.
  • API endpoint = địa chỉ cụ thể của một chức năng. Ví dụ: GET /api/users lấy danh sách user, POST /api/users tạo user mới.
  • Response trả về JSON — không phải HTML, không có giao diện. Frontend nhận JSON rồi tự render ra UI.
  • Trong vibe coding: Claude Code generate app/api/ routes là nó đang tạo API endpoint cho bạn.
Bỏ qua đượcGraphQL, WebSocket, gRPC, tự viết API documentation.

E3
Authentication — Đăng nhập hoạt động thế nào?
Khái niệm nền
▼
Analogy cho designerĐăng nhập lần đầu như nhận thẻ VIP ở cửa quán — nhân viên check tên bạn trong danh sách, đưa thẻ. Mỗi lần vào sau, bạn chỉ cần giơ thẻ — không cần xuất trình CMND lại. Token hoạt động đúng như vậy.
Flow — JWT Authentication
Nội dung chính
  • Authentication = xác minh bạn là ai (đăng nhập). Authorization = kiểm tra bạn có quyền làm gì (phân quyền).
  • JWT (JSON Web Token) = token nhỏ gọn chứa thông tin user, có chữ ký số. Server tạo ra, client giữ và gửi kèm mỗi request.
  • Token thường lưu trong cookie hoặc localStorage của browser.
  • Token có thời hạn (expiry) — hết hạn phải đăng nhập lại hoặc dùng refresh token.
  • Trong vibe coding: dùng NextAuth.js hoặc Supabase Auth — Claude Code setup giúp bạn, bạn chỉ cần hiểu flow này để debug khi lỗi 401.
Bỏ qua đượcOAuth flow chi tiết, refresh token rotation, tự implement JWT từ đầu — dùng thư viện có sẵn.
